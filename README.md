# Vanguard – Wildlife Conservation & Anti-Poaching Intelligence Platform

AI-powered ranger operations dashboard combining **camera trap vision**, **acoustic detection**, and **community reporting** for protected areas. Built for Microsoft Hackathon.

- **Frontend:** React, TypeScript, Tailwind, Vite, Leaflet (CartoDB dark)
- **Backend:** Node.js, Express
- **Run:** `start-all.bat` (launches frontend + backend)

---

## Making the demo as realistic as possible (APIs & keys)

The app uses **real APIs** where keys are provided and **deterministic demo data** otherwise, so you can show a credible, production-style demo at the hackathon.

### 1. Live data (no key required)

- **Weather & environment** – [Open-Meteo](https://open-meteo.com) (free, no key). When the backend can reach it, the dashboard shows **“LIVE (Open-Meteo)”** in the Environment panel.
- **Biodiversity** – [GBIF](https://www.gbif.org) occurrence API (no key) for species near park coordinates.
- **Satellite events** – [NASA EONET](https://eonet.gsfc.nasa.gov) (no key) for nearby events.

### 2. Live AI – optional keys (set in environment before starting the backend)

| Purpose | Env variable | What it does |
|--------|---------------|--------------|
| **Image / species classification** | `CLARIFAI_PAT` | Uses [Clarifai](https://www.clarifai.com) for uploaded camera trap images. Tried first if set. |
| **Image / species classification** | `HF_TOKEN` | [Hugging Face](https://huggingface.co/settings/tokens) Inference API (e.g. `google/vit-base-patch16-224`). Used if Clarifai is not set or fails. |
| **AI-generated recommendations** | `OPENROUTER_API_KEY` | [Open Router](https://openrouter.ai) for one-line ranger recommendations (alerts + sound analysis). Uses a small GPT-style model so recommendations feel real in the demo. |

**How to run with keys (Windows, in the backend folder):**

```bat
set CLARIFAI_PAT=your_clarifai_key
set HF_TOKEN=your_huggingface_token
set OPENROUTER_API_KEY=your_openrouter_key
node server.js
```

Or use a `.env` file in `backend/` and load it (e.g. with `dotenv`) so you don’t commit keys.

### 3. Realistic data already in the app

- **Fauna catalog** – Species names, scientific names, and **IUCN-style status** (EN, VU, CR, NT, LC). Seed data includes short **citations** (e.g. NTCA 2022, IUCN 2024, SANParks) so judges can see the numbers are grounded in real sources.
- **Alerts and acoustic events** – Descriptions and labels match real-world ranger/operational wording (e.g. “High-caliber rifle suspected”, “Chainsaw-like harmonic pattern”).
- **Correlation engine** – Zone + multi-source rules (e.g. gunshot + human presence + vehicle) produce **ELEVATED / HIGH / CRITICAL** incidents with consistent, actionable text.

### 4. What “live” looks like in the UI

- **Environment panel** – “LIVE (Open-Meteo)” when weather is from the API; “OFFLINE” when using fallback.
- **Species ID (upload image)** – If `CLARIFAI_PAT` or `HF_TOKEN` is set and the request succeeds, the result is from a real vision API; otherwise a deterministic, species-appropriate fallback is used.
- **Sound analysis** – Classification is deterministic by sample type. If `OPENROUTER_API_KEY` is set, the **recommended action** is generated by the AI; the modal shows “Recommendation generated by Open Router (live AI).”

With **Open-Meteo + one vision key (Clarifai or HF) + Open Router**, the demo uses live weather, live image classification, and AI-generated recommendations, which is enough to show that the system is **realistic and implementable in the real world**.

## Will the APIs work when hosted (e.g. on Render)?

**Yes**, with a few setup steps:

- **Outbound HTTPS** – Render (and similar platforms) allow the server to call external APIs. All of these are plain HTTPS from your backend to:
  - `api.open-meteo.com` (weather)
  - `api.gbif.org` (biodiversity)
  - `eonet.gsfc.nasa.gov` (NASA events)
  - `api.clarifai.com`, `api-inference.huggingface.co`, `openrouter.ai` (when you set the keys)
- **Node 18+** – The backend uses the built-in `fetch`. The repo sets `"engines": { "node": ">=18.0.0" }` so Render picks a Node version that has `fetch`. No extra packages needed.
- **Env vars** – In Render’s dashboard, set `CLARIFAI_PAT`, `HF_TOKEN`, `OPENROUTER_API_KEY` (and `PORT` if required). If unset, the app falls back to deterministic demo behaviour and still runs.
- **Persistence** – Fauna catalog, spottings, and audio events are stored in `backend/data/*.json` on disk. On Render’s **ephemeral filesystem**, these files are lost on redeploy or restart. For a hackathon demo that’s usually acceptable (seed data repopulates on first request). For a permanent deployment you’d switch to a database (e.g. Render PostgreSQL) or object storage.
- **Timeouts** – External API calls are short (seconds). Stay within your plan’s request timeout (e.g. 30 s on free tier).

**Deploy steps (Render):** Use the backend folder as the service root, set **Start Command** to `npm start` (runs `node server.js`), add env vars, and ensure the built frontend is in `frontend/dist` (build the frontend before deploy and commit `dist`, or add a build step that runs `npm run build` in the frontend and copies `dist` into the backend tree so the server can serve it).
